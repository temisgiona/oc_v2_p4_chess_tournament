8 r"""
9 .. dialect:: mssql+pyodbc
10 :name: PyODBC
11 :dbapi: pyodbc
12 :connectstring: mssql+pyodbc://<username>:<password>@<dsnname>
13 :url: http://pypi.python.org/pypi/pyodbc/
14  
15 Connecting to PyODBC
16 --------------------
17  
18 The URL here is to be translated to PyODBC connection strings, as
19 detailed in `ConnectionStrings <https://code.google.com/p/pyodbc/wiki/ConnectionStrings>`_.
20  
21 DSN Connections
22 ^^^^^^^^^^^^^^^
23  
24 A DSN connection in ODBC means that a pre-existing ODBC datasource is
25 configured on the client machine. The application then specifies the name
26 of this datasource, which encompasses details such as the specific ODBC driver
27 in use as well as the network address of the database. Assuming a datasource
28 is configured on the client, a basic DSN-based connection looks like::
29  
30 engine = create_engine("mssql+pyodbc://scott:tiger@some_dsn")
31  
32 Which above, will pass the following connection string to PyODBC::
33  
34 dsn=mydsn;UID=user;PWD=pass
35  
36 If the username and password are omitted, the DSN form will also add
37 the ``Trusted_Connection=yes`` directive to the ODBC string.
38  
39 Hostname Connections
40 ^^^^^^^^^^^^^^^^^^^^
41  
42 Hostname-based connections are also supported by pyodbc. These are often
43 easier to use than a DSN and have the additional advantage that the specific
44 database name to connect towards may be specified locally in the URL, rather
45 than it being fixed as part of a datasource configuration.
46  
47 When using a hostname connection, the driver name must also be specified in the
48 query parameters of the URL. As these names usually have spaces in them, the
49 name must be URL encoded which means using plus signs for spaces::
50  
51 engine = create_engine("mssql+pyodbc://scott:tiger@myhost:port/databasename?driver=SQL+Server+Native+Client+10.0")
52  
53 Other keywords interpreted by the Pyodbc dialect to be passed to
54 ``pyodbc.connect()`` in both the DSN and hostname cases include:
55 ``odbc_autotranslate``, ``ansi``, ``unicode_results``, ``autocommit``,
56 ``authentication`` (e.g., ``authentication=ActiveDirectoryIntegrated``).
57 Note that in order for the dialect to recognize these keywords
58 (including the ``driver`` keyword above) they must be all lowercase.
59  
60 Pass through exact Pyodbc string
61 ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
62  
63 A PyODBC connection string can also be sent in pyodbc's format directly, as
64 specified in `ConnectionStrings
65 <https://code.google.com/p/pyodbc/wiki/ConnectionStrings>`_ into the driver
66 using the parameter ``odbc_connect``. The delimeters must be URL encoded, as
67 illustrated below using ``urllib.parse.quote_plus``::
68  
69 import urllib
70 params = urllib.parse.quote_plus("DRIVER={SQL Server Native Client 10.0};SERVER=dagger;DATABASE=test;UID=user;PWD=password")
71  
72 engine = create_engine("mssql+pyodbc:///?odbc_connect=%s" % params)
73  
74 Pyodbc Pooling / connection close behavior
75 ------------------------------------------
76  
77 PyODBC uses internal `pooling
78 <https://github.com/mkleehammer/pyodbc/wiki/The-pyodbc-Module#pooling>`_ by
79 default, which means connections will be longer lived than they are within
80 SQLAlchemy itself. As SQLAlchemy has its own pooling behavior, it is often
81 preferable to disable this behavior. This behavior can only be disabled
82 globally at the PyODBC module level, **before** any connections are made::
83  
84 import pyodbc
85  
86 pyodbc.pooling = False
87  
88 # don't use the engine before pooling is set to False
89 engine = create_engine("mssql+pyodbc://user:pass@dsn")
90  
91 If this variable is left at its default value of ``True``, **the application
92 will continue to maintain active database connections**, even when the
93 SQLAlchemy engine itself fully discards a connection or if the engine is
94 disposed.
95  
96 .. seealso::
97  
98 `pooling <https://github.com/mkleehammer/pyodbc/wiki/The-pyodbc-Module#pooling>`_ -
99 in the PyODBC documentation.
100  
101 Driver / Unicode Support
102 -------------------------
103  
104 PyODBC works best with Microsoft ODBC drivers, particularly in the area
105 of Unicode support on both Python 2 and Python 3.
106  
107 Using the FreeTDS ODBC drivers on Linux or OSX with PyODBC is **not**
108 recommended; there have been historically many Unicode-related issues
109 in this area, including before Microsoft offered ODBC drivers for Linux
110 and OSX. Now that Microsoft offers drivers for all platforms, for
111 PyODBC support these are recommended. FreeTDS remains relevant for
112 non-ODBC drivers such as pymssql where it works very well.
113  
114  
115 Rowcount Support
116 ----------------
117  
118 Pyodbc only has partial support for rowcount. See the notes at
119 :ref:`mssql_rowcount_versioning` for important notes when using ORM
120 versioning.
121  
122 .. _mssql_pyodbc_fastexecutemany:
123  
124 Fast Executemany Mode
125 ---------------------
126  
127 The Pyodbc driver has added support for a "fast executemany" mode of execution
128 which greatly reduces round trips for a DBAPI ``executemany()`` call when using
129 Microsoft ODBC drivers, for **limited size batches that fit in memory**. The
130 feature is enabled by setting the flag ``.fast_executemany`` on the DBAPI
131 cursor when an executemany call is to be used. The SQLAlchemy pyodbc SQL
132 Server dialect supports setting this flag automatically when the
133 ``.fast_executemany`` flag is passed to
134 :func:`_sa.create_engine` ; note that the ODBC driver must be the Microsoft
135 driver in order to use this flag::
136  
137 engine = create_engine(
138 "mssql+pyodbc://scott:tiger@mssql2017:1433/test?driver=ODBC+Driver+13+for+SQL+Server",
139 fast_executemany=True)
140  
141 .. warning:: The pyodbc fast_executemany mode **buffers all rows in memory** and is
142 not compatible with very large batches of data. A future version of SQLAlchemy
143 may support this flag as a per-execution option instead.
144  
145 .. versionadded:: 1.3
146  
147 .. seealso::
148  
149 `fast executemany <https://github.com/mkleehammer/pyodbc/wiki/Features-beyond-the-DB-API#fast_executemany>`_
150 - on github

174  
175 The routines here are needed for older pyodbc versions
176 as well as current mxODBC versions.

188 if self.asdecimal and isinstance(value, decimal.Decimal):
189 adjusted = value.adjusted()

191 return self._small_dec_to_string(value)
192 elif adjusted > 7:

243 class _ms_binary_pyodbc(object):
244 """Wraps binary values in dialect-specific Binary wrapper.
245 If the value is null, return a pyodbc-specific BinaryNull
246 object to prevent pyODBC [and FreeTDS] from defaulting binary
247 NULL types to SQLWCHAR and causing implicit conversion errors.
248 """
249  
250 def bind_processor(self, dialect):
251 if dialect.dbapi is None:
252 return None
253  
254 DBAPIBinary = dialect.dbapi.Binary
255  
256 def process(value):
257 if value is not None:
258 return DBAPIBinary(value)
259 else:
260 # pyodbc-specific
261 return dialect.dbapi.BinaryNull
262  
263 return process

266 class _ODBCDateTimeOffset(DATETIMEOFFSET):
267 def bind_processor(self, dialect):
268 def process(value):
269 if value is None:
270 return None
271 elif isinstance(value, util.string_types):
272 # if a string was passed directly, allow it through
273 return value
274 else:
275 # Convert to string format required by T-SQL
276 dto_string = value.strftime("%Y-%m-%d %H:%M:%S.%f %z")
277 # offset needs a colon, e.g., -0700 -> -07:00
278 # "UTC offset in the form (+-)HHMM[SS[.ffffff]]"
279 # backend currently rejects seconds / fractional seconds
280 dto_string = re.sub(
281 r"([\+\-]\d{2})([\d\.]+)$", r"\1:\2", dto_string
282 )
283 return dto_string
284  
285 return process

297 _embedded_scope_identity = False
298  
299 def pre_exec(self):
300 """where appropriate, issue "select scope_identity()" in the same
301 statement.
302  
303 Background on why "scope_identity()" is preferable to "@@identity":
304 http://msdn.microsoft.com/en-us/library/ms190315.aspx
305  
306 Background on why we attempt to embed "scope_identity()" into the same
307 statement as the INSERT:
308 http://code.google.com/p/pyodbc/wiki/FAQs#How_do_I_retrieve_autogenerated/identity_values?
309  
310 """
311  
312 super(MSExecutionContext_pyodbc, self).pre_exec()
313  
314 # don't embed the scope_identity select into an
315 # "INSERT .. DEFAULT VALUES"
316 if (
317 self._select_lastrowid
318 and self.dialect.use_scope_identity
319 and len(self.parameters[0])
320 ):
321 self._embedded_scope_identity = True
322  
323 self.statement += "; select scope_identity()"
324  
325 def post_exec(self):
326 if self._embedded_scope_identity:
327 # Fetch the last inserted id from the manipulated statement
328 # We may have to skip over a number of result sets with
329 # no data (due to triggers, etc.)
330 while True:
331 try:
332 # fetchall() ensures the cursor is consumed
333 # without closing it (FreeTDS particularly)
334 row = self.cursor.fetchall()[0]
335 break
336 except self.dialect.dbapi.Error:
337 # no way around this - nextset() consumes the previous set
338 # so we need to just keep flipping
339 self.cursor.nextset()
340  
341 self._lastrowid = int(row[0])
342 else:
343 super(MSExecutionContext_pyodbc, self).post_exec()

387 def _get_server_version_info(self, connection):
388 try:
389 # "Version of the instance of SQL Server, in the form
390 # of 'major.minor.build.revision'"
391 raw = connection.scalar(
392 "SELECT CAST(SERVERPROPERTY('ProductVersion') AS VARCHAR)"
393 )
394 except exc.DBAPIError:
395 # SQL Server docs indicate this function isn't present prior to
396 # 2008. Before we had the VARCHAR cast above, pyodbc would also
397 # fail on this query.
398 return super(MSDialect_pyodbc, self)._get_server_version_info(
399 connection, allow_chars=False
400 )
401 else:
402 version = []
403 r = re.compile(r"[.\-]")
404 for n in r.split(raw):
405 try:
406 version.append(int(n))
407 except ValueError:
408 pass
409 return tuple(version)
