16 Auto Increment Behavior
17 -----------------------
18  
19 SQLAlchemy Table objects which include integer primary keys are usually
20 assumed to have "autoincrementing" behavior, meaning they can generate their
21 own primary key values upon INSERT. Since Oracle has no "autoincrement"
22 feature, SQLAlchemy relies upon sequences to produce these values. With the
23 Oracle dialect, *a sequence must always be explicitly specified to enable
24 autoincrement*. This is divergent with the majority of documentation
25 examples which assume the usage of an autoincrement-capable database. To
26 specify sequences, use the sqlalchemy.schema.Sequence object which is passed
27 to a Column construct::
28  
29 t = Table('mytable', metadata,
30 Column('id', Integer, Sequence('id_seq'), primary_key=True),
31 Column(...), ...
32 )
33  
34 This step is also required when using table reflection, i.e. autoload=True::
35  
36 t = Table('mytable', metadata,
37 Column('id', Integer, Sequence('id_seq'), primary_key=True),
38 autoload=True
39 )

42 .. _oracle_isolation_level:

44 Transaction Isolation Level / Autocommit
45 ----------------------------------------
46  
47 The Oracle database supports "READ COMMITTED" and "SERIALIZABLE" modes of
48 isolation. The AUTOCOMMIT isolation level is also supported by the cx_Oracle
49 dialect.
50  
51 To set using per-connection execution options::
52  
53 connection = engine.connect()
54 connection = connection.execution_options(
55 isolation_level="AUTOCOMMIT"
56 )
57  
58 For ``READ COMMITTED`` and ``SERIALIZABLE``, the Oracle dialect sets the
59 level at the session level using ``ALTER SESSION``, which is reverted back
60 to its default setting when the connection is returned to the connection
61 pool.
62  
63 Valid values for ``isolation_level`` include:
64  
65 * ``READ COMMITTED``
66 * ``AUTOCOMMIT``
67 * ``SERIALIZABLE``
68  
69 .. note:: The implementation for the
70 :meth:`_engine.Connection.get_isolation_level` method as implemented by the
71 Oracle dialect necessarily forces the start of a transaction using the
72 Oracle LOCAL_TRANSACTION_ID function; otherwise no level is normally
73 readable.
74  
75 Additionally, the :meth:`_engine.Connection.get_isolation_level` method will
76 raise an exception if the ``v$transaction`` view is not available due to
77 permissions or other reasons, which is a common occurrence in Oracle
78 installations.
79  
80 The cx_Oracle dialect attempts to call the
81 :meth:`_engine.Connection.get_isolation_level` method when the dialect makes
82 its first connection to the database in order to acquire the
83 "default"isolation level. This default level is necessary so that the level
84 can be reset on a connection after it has been temporarily modified using
85 :meth:`_engine.Connection.execution_options` method. In the common event
86 that the :meth:`_engine.Connection.get_isolation_level` method raises an
87 exception due to ``v$transaction`` not being readable as well as any other
88 database-related failure, the level is assumed to be "READ COMMITTED". No
89 warning is emitted for this initial first-connect condition as it is
90 expected to be a common restriction on Oracle databases.
91  
92 .. versionadded:: 1.3.16 added support for AUTOCOMMIT to the cx_oracle dialect
93 as well as the notion of a default isolation level
94  
95 .. versionadded:: 1.3.21 Added support for SERIALIZABLE as well as live
96 reading of the isolation level.
97  
98 .. versionchanged:: 1.3.22 In the event that the default isolation
99 level cannot be read due to permissions on the v$transaction view as
100 is common in Oracle installations, the default isolation level is hardcoded
101 to "READ COMMITTED" which was the behavior prior to 1.3.21.

107 Identifier Casing
108 -----------------
109  
110 In Oracle, the data dictionary represents all case insensitive identifier
111 names using UPPERCASE text. SQLAlchemy on the other hand considers an
112 all-lower case identifier name to be case insensitive. The Oracle dialect
113 converts all case insensitive identifiers to and from those two formats during
114 schema level communication, such as reflection of tables and indexes. Using
115 an UPPERCASE name on the SQLAlchemy side indicates a case sensitive
116 identifier, and SQLAlchemy will quote the name - this will cause mismatches
117 against data dictionary data received from Oracle, so unless identifier names
118 have been truly created as case sensitive (i.e. using quoted names), all
119 lowercase names should be used on the SQLAlchemy side.

121 .. _oracle_max_identifier_lengths:

123 Max Identifier Lengths
124 ----------------------
125  
126 Oracle has changed the default max identifier length as of Oracle Server
127 version 12.2. Prior to this version, the length was 30, and for 12.2 and
128 greater it is now 128. This change impacts SQLAlchemy in the area of
129 generated SQL label names as well as the generation of constraint names,
130 particularly in the case where the constraint naming convention feature
131 described at :ref:`constraint_naming_conventions` is being used.
132  
133 To assist with this change and others, Oracle includes the concept of a
134 "compatibility" version, which is a version number that is independent of the
135 actual server version in order to assist with migration of Oracle databases,
136 and may be configured within the Oracle server itself. This compatibility
137 version is retrieved using the query ``SELECT value FROM v$parameter WHERE
138 name = 'compatible';``. The SQLAlchemy Oracle dialect, when tasked with
139 determining the default max identifier length, will attempt to use this query
140 upon first connect in order to determine the effective compatibility version of
141 the server, which determines what the maximum allowed identifier length is for
142 the server. If the table is not available, the server version information is
143 used instead.
144  
145 For the duration of the SQLAlchemy 1.3 series, the default max identifier
146 length will remain at 30, even if compatibility version 12.2 or greater is in
147 use. When the newer version is detected, a warning will be emitted upon first
148 connect, which refers the user to make use of the
149 :paramref:`_sa.create_engine.max_identifier_length`
150 parameter in order to assure
151 forwards compatibility with SQLAlchemy 1.4, which will be changing this value
152 to 128 when compatibility version 12.2 or greater is detected.
153  
154 Using :paramref:`_sa.create_engine.max_identifier_length`,
155 the effective identifier
156 length used by the SQLAlchemy dialect will be used as given, overriding the
157 current default value of 30, so that when Oracle 12.2 or greater is used, the
158 newer identifier length may be taken advantage of::
159  
160 engine = create_engine(
161 "oracle+cx_oracle://scott:tiger@oracle122",
162 max_identifier_length=128)
163  
164 The maximum identifier length comes into play both when generating anonymized
165 SQL labels in SELECT statements, but more crucially when generating constraint
166 names from a naming convention. It is this area that has created the need for
167 SQLAlchemy to change this default conservatively. For example, the following
168 naming convention produces two very different constraint names based on the
169 identifier length::
170  
171 from sqlalchemy import Column
172 from sqlalchemy import Index
173 from sqlalchemy import Integer
174 from sqlalchemy import MetaData
175 from sqlalchemy import Table
176 from sqlalchemy.dialects import oracle
177 from sqlalchemy.schema import CreateIndex
178  
179 m = MetaData(naming_convention={"ix": "ix_%(column_0N_name)s"})
180  
181 t = Table(
182 "t",
183 m,
184 Column("some_column_name_1", Integer),
185 Column("some_column_name_2", Integer),
186 Column("some_column_name_3", Integer),
187 )
188  
189 ix = Index(
190 None,
191 t.c.some_column_name_1,
192 t.c.some_column_name_2,
193 t.c.some_column_name_3,
194 )
195  
196 oracle_dialect = oracle.dialect(max_identifier_length=30)
197 print(CreateIndex(ix).compile(dialect=oracle_dialect))
198  
199 With an identifier length of 30, the above CREATE INDEX looks like::
200  
201 CREATE INDEX ix_some_column_name_1s_70cd ON t
202 (some_column_name_1, some_column_name_2, some_column_name_3)
203  
204 However with length=128, it becomes::
205  
206 CREATE INDEX ix_some_column_name_1some_column_name_2some_column_name_3 ON t
207 (some_column_name_1, some_column_name_2, some_column_name_3)
208  
209 The implication here is that by upgrading SQLAlchemy to version 1.4 on
210 an existing Oracle 12.2 or greater database, the generation of constraint
211 names will change, which can impact the behavior of database migrations.
212 A key example is a migration that wishes to "DROP CONSTRAINT" on a name that
213 was previously generated with the shorter length. This migration will fail
214 when the identifier length is changed without the name of the index or
215 constraint first being adjusted.
216  
217 Therefore, applications are strongly advised to make use of
218 :paramref:`_sa.create_engine.max_identifier_length`
219 in order to maintain control
220 of the generation of truncated names, and to fully review and test all database
221 migrations in a staging environment when changing this value to ensure that the
222 impact of this change has been mitigated.
223  
224  
225 .. versionadded:: 1.3.9 Added the
226 :paramref:`_sa.create_engine.max_identifier_length` parameter; the Oracle
227 dialect now detects compatibility version 12.2 or greater and warns
228 about upcoming max identitifier length changes in SQLAlchemy 1.4.

231 LIMIT/OFFSET Support
232 --------------------
233  
234 Oracle has no support for the LIMIT or OFFSET keywords. SQLAlchemy uses
235 a wrapped subquery approach in conjunction with ROWNUM. The exact methodology
236 is taken from
237 http://www.oracle.com/technetwork/issue-archive/2006/06-sep/o56asktom-086197.html .
238  
239 There are two options which affect its behavior:
240  
241 * the "FIRST ROWS()" optimization keyword is not used by default. To enable
242 the usage of this optimization directive, specify ``optimize_limits=True``
243 to :func:`_sa.create_engine`.
244 * the values passed for the limit/offset are sent as bound parameters. Some
245 users have observed that Oracle produces a poor query plan when the values
246 are sent as binds and not rendered literally. To render the limit/offset
247 values literally within the SQL statement, specify
248 ``use_binds_for_limits=False`` to :func:`_sa.create_engine`.
249  
250 Some users have reported better performance when the entirely different
251 approach of a window query is used, i.e. ROW_NUMBER() OVER (ORDER BY), to
252 provide LIMIT/OFFSET (note that the majority of users don't observe this).
253 To suit this case the method used for LIMIT/OFFSET can be replaced entirely.
254 See the recipe at
255 http://www.sqlalchemy.org/trac/wiki/UsageRecipes/WindowFunctionsByDefault
256 which installs a select compiler that overrides the generation of limit/offset
257 with a window function.

259 .. _oracle_returning:
260  
261 RETURNING Support
262 -----------------
263  
264 The Oracle database supports a limited form of RETURNING, in order to retrieve
265 result sets of matched rows from INSERT, UPDATE and DELETE statements.
266 Oracle's RETURNING..INTO syntax only supports one row being returned, as it
267 relies upon OUT parameters in order to function. In addition, supported
268 DBAPIs have further limitations (see :ref:`cx_oracle_returning`).
269  
270 SQLAlchemy's "implicit returning" feature, which employs RETURNING within an
271 INSERT and sometimes an UPDATE statement in order to fetch newly generated
272 primary key values and other SQL defaults and expressions, is normally enabled
273 on the Oracle backend. By default, "implicit returning" typically only
274 fetches the value of a single ``nextval(some_seq)`` expression embedded into
275 an INSERT in order to increment a sequence within an INSERT statement and get
276 the value back at the same time. To disable this feature across the board,
277 specify ``implicit_returning=False`` to :func:`_sa.create_engine`::
278  
279 engine = create_engine("oracle://scott:tiger@dsn",
280 implicit_returning=False)
281  
282 Implicit returning can also be disabled on a table-by-table basis as a table
283 option::
284  
285 # Core Table
286 my_table = Table("my_table", metadata, ..., implicit_returning=False)
287  
288  
289 # declarative
290 class MyClass(Base):
291 __tablename__ = 'my_table'
292 __table_args__ = {"implicit_returning": False}
293  
294 .. seealso::
295  
296 :ref:`cx_oracle_returning` - additional cx_oracle-specific restrictions on
297 implicit returning.

299 ON UPDATE CASCADE
300 -----------------
301  
302 Oracle doesn't have native ON UPDATE CASCADE functionality. A trigger based
303 solution is available at
304 http://asktom.oracle.com/tkyte/update_cascade/index.html .

306 When using the SQLAlchemy ORM, the ORM has limited ability to manually issue

308 "deferrable=True, initially='deferred'" keyword arguments,

311 Oracle 8 Compatibility
312 ----------------------
313  
314 When Oracle 8 is detected, the dialect internally configures itself to the
315 following behaviors:
316  
317 * the use_ansi flag is set to False. This has the effect of converting all
318 JOIN phrases into the WHERE clause, and in the case of LEFT OUTER JOIN
319 makes use of Oracle's (+) operator.
320  
321 * the NVARCHAR2 and NCLOB datatypes are no longer generated as DDL when
322 the :class:`~sqlalchemy.types.Unicode` is used - VARCHAR2 and CLOB are
323 issued instead. This because these types don't seem to work correctly on
324 Oracle 8 even though they are available. The
325 :class:`~sqlalchemy.types.NVARCHAR` and
326 :class:`~sqlalchemy.dialects.oracle.NCLOB` types will always generate
327 NVARCHAR2 and NCLOB.
328  
329 * the "native unicode" mode is disabled when using cx_oracle, i.e. SQLAlchemy
330 encodes all Python unicode objects to "string" before passing in as bind
331 parameters.

333 Synonym/DBLINK Reflection
334 -------------------------
335  
336 When using reflection with Table objects, the dialect can optionally search
337 for tables indicated by synonyms, either in local or remote schemas or
338 accessed over DBLINK, by passing the flag ``oracle_resolve_synonyms=True`` as
339 a keyword argument to the :class:`_schema.Table` construct::
340  
341 some_table = Table('some_table', autoload=True,
342 autoload_with=some_engine,
343 oracle_resolve_synonyms=True)
344  
345 When this flag is set, the given name (such as ``some_table`` above) will
346 be searched not just in the ``ALL_TABLES`` view, but also within the
347 ``ALL_SYNONYMS`` view to see if this name is actually a synonym to another
348 name. If the synonym is located and refers to a DBLINK, the oracle dialect
349 knows how to locate the table's information using DBLINK syntax(e.g.
350 ``@dblink``).
351  
352 ``oracle_resolve_synonyms`` is accepted wherever reflection arguments are
353 accepted, including methods such as :meth:`_schema.MetaData.reflect` and
354 :meth:`_reflection.Inspector.get_columns`.
355  
356 If synonyms are not in use, this flag should be left disabled.

358 .. _oracle_constraint_reflection:
359  
360 Constraint Reflection
361 ---------------------
362  
363 The Oracle dialect can return information about foreign key, unique, and
364 CHECK constraints, as well as indexes on tables.
365  
366 Raw information regarding these constraints can be acquired using
367 :meth:`_reflection.Inspector.get_foreign_keys`,
368 :meth:`_reflection.Inspector.get_unique_constraints`,
369 :meth:`_reflection.Inspector.get_check_constraints`, and
370 :meth:`_reflection.Inspector.get_indexes`.
371  
372 .. versionchanged:: 1.2 The Oracle dialect can now reflect UNIQUE and
373 CHECK constraints.
374  
375 When using reflection at the :class:`_schema.Table` level, the
376 :class:`_schema.Table`
377 will also include these constraints.
378  
379 Note the following caveats:
380  
381 * When using the :meth:`_reflection.Inspector.get_check_constraints` method,
382 Oracle
383 builds a special "IS NOT NULL" constraint for columns that specify
384 "NOT NULL". This constraint is **not** returned by default; to include
385 the "IS NOT NULL" constraints, pass the flag ``include_all=True``::
386  
387 from sqlalchemy import create_engine, inspect
388  
389 engine = create_engine("oracle+cx_oracle://s:t@dsn")
390 inspector = inspect(engine)
391 all_check_constraints = inspector.get_check_constraints(
392 "some_table", include_all=True)
393  
394 * in most cases, when reflecting a :class:`_schema.Table`,
395 a UNIQUE constraint will
396 **not** be available as a :class:`.UniqueConstraint` object, as Oracle
397 mirrors unique constraints with a UNIQUE index in most cases (the exception
398 seems to be when two or more unique constraints represent the same columns);
399 the :class:`_schema.Table` will instead represent these using
400 :class:`.Index`
401 with the ``unique=True`` flag set.
402  
403 * Oracle creates an implicit index for the primary key of a table; this index
404 is **excluded** from all index results.
405  
406 * the list of columns reflected for an index will not include column names
407 that start with SYS_NC.
408  
409 Table names with SYSTEM/SYSAUX tablespaces
410 -------------------------------------------
411  
412 The :meth:`_reflection.Inspector.get_table_names` and
413 :meth:`_reflection.Inspector.get_temp_table_names`
414 methods each return a list of table names for the current engine. These methods
415 are also part of the reflection which occurs within an operation such as
416 :meth:`_schema.MetaData.reflect`. By default,
417 these operations exclude the ``SYSTEM``
418 and ``SYSAUX`` tablespaces from the operation. In order to change this, the
419 default list of tablespaces excluded can be changed at the engine level using
420 the ``exclude_tablespaces`` parameter::
421  
422 # exclude SYSAUX and SOME_TABLESPACE, but not SYSTEM
423 e = create_engine(
424 "oracle://scott:tiger@xe",
425 exclude_tablespaces=["SYSAUX", "SOME_TABLESPACE"])
426

429 DateTime Compatibility
430 ----------------------
431  
432 Oracle has no datatype known as ``DATETIME``, it instead has only ``DATE``,
433 which can actually store a date and time value. For this reason, the Oracle
434 dialect provides a type :class:`_oracle.DATE` which is a subclass of
435 :class:`.DateTime`. This type has no special behavior, and is only
436 present as a "marker" for this type; additionally, when a database column
437 is reflected and the type is reported as ``DATE``, the time-supporting
438 :class:`_oracle.DATE` type is used.
439  
440 .. versionchanged:: 0.9.4 Added :class:`_oracle.DATE` to subclass
441 :class:`.DateTime`. This is a change as previous versions
442 would reflect a ``DATE`` column as :class:`_types.DATE`, which subclasses
443 :class:`.Date`. The only significance here is for schemes that are
444 examining the type of column for use in special Python translations or
445 for migrating schemas to other database backends.

449 Oracle Table Options
450 -------------------------
451  
452 The CREATE TABLE phrase supports the following options with Oracle
453 in conjunction with the :class:`_schema.Table` construct:
454  
455  
456 * ``ON COMMIT``::
457  
458 Table(
459 "some_table", metadata, ...,
460 prefixes=['GLOBAL TEMPORARY'], oracle_on_commit='PRESERVE ROWS')
461  
462 .. versionadded:: 1.0.0
463  
464 * ``COMPRESS``::
465  
466 Table('mytable', metadata, Column('data', String(32)),
467 oracle_compress=True)
468  
469 Table('mytable', metadata, Column('data', String(32)),
470 oracle_compress=6)
471

473 level, or ``True`` to use the default compression level.

477 .. _oracle_index_options:
478  
479 Oracle Specific Index Options
480 -----------------------------
481  
482 Bitmap Indexes
483 ~~~~~~~~~~~~~~
484  
485 You can specify the ``oracle_bitmap`` parameter to create a bitmap index
486 instead of a B-tree index::
487  
488 Index('my_index', my_table.c.data, oracle_bitmap=True)
489  
490 Bitmap indexes cannot be unique and cannot be compressed. SQLAlchemy will not
491 check for such limitations, only the database will.
492  
493 .. versionadded:: 1.0.0
494  
495 Index compression
496 ~~~~~~~~~~~~~~~~~
497  
498 Oracle has a more efficient storage mode for indexes containing lots of
499 repeated values. Use the ``oracle_compress`` parameter to turn on key
500 compression::
501  
502 Index('my_index', my_table.c.data, oracle_compress=True)
503  
504 Index('my_index', my_table.c.data1, my_table.c.data2, unique=True,
505 oracle_compress=1)
506  
507 The ``oracle_compress`` parameter accepts either an integer specifying the
508 number of prefix columns to compress, or ``True`` to use the default (all
509 columns for non-unique indexes, all but the last column for unique indexes).

1021 def visit_select(self, select, **kwargs):

1023 so tries to wrap it in a subquery with ``rownum`` criterion.
1024 """
1025

1029 select, kwargs.get("asfrom", False)

1039 # See http://www.oracle.com/technology/oramag/oracle/06-sep/\
1040 # o56asktom.html
1041 #
1042 # Generalized form of an Oracle pagination query:
1043 # select ... from (
1044 # select /*+ FIRST_ROWS(N) */ ...., rownum as ora_rn from
1045 # ( select distinct ... where ... order by ...

1063 "/*+ FIRST_ROWS(%d) */" % select._limit

1070 for_update = select._for_update_arg

1073 for_update._copy_internals()

1084 if limit_clause is not None:

1086 # use simple int limits, will raise an exception
1087 # if the limit isn't specified this way
1088 max_row = select._limit
1089  
1090 if offset_clause is not None:

1093 else:
1094 max_row = limit_clause

1104 select = limitselect
1105 else:

1116 offsetselect._is_wrapper = True

1119 for elem in for_update.of:

1121 limitselect.append_column(elem)

1125 "%d" % select._offset

1128 sql.literal_column("ora_rn") > offset_clause
