46 class WebSocketClientProtocol(WebSocketCommonProtocol):
47 """
48 :class:`~asyncio.Protocol` subclass implementing a WebSocket client.
49  
50 :class:`WebSocketClientProtocol`:
51  
52 * performs the opening handshake to establish the connection;
53 * provides :meth:`recv` and :meth:`send` coroutines for receiving and
54 sending messages;
55 * deals with control frames automatically;
56 * performs the closing handshake to terminate the connection.
57  
58 :class:`WebSocketClientProtocol` supports asynchronous iteration::
59  
60 async for message in websocket:
61 await process(message)
62  
63 The iterator yields incoming messages. It exits normally when the
64 connection is closed with the close code 1000 (OK) or 1001 (going away).
65 It raises a :exc:`~websockets.exceptions.ConnectionClosedError` exception
66 when the connection is closed with any other code.
67  
68 Once the connection is open, a `Ping frame`_ is sent every
69 ``ping_interval`` seconds. This serves as a keepalive. It helps keeping
70 the connection open, especially in the presence of proxies with short
71 timeouts on inactive connections. Set ``ping_interval`` to ``None`` to
72 disable this behavior.
73  
74 .. _Ping frame: https://tools.ietf.org/html/rfc6455#section-5.5.2
75  
76 If the corresponding `Pong frame`_ isn't received within ``ping_timeout``
77 seconds, the connection is considered unusable and is closed with
78 code 1011. This ensures that the remote endpoint remains responsive. Set
79 ``ping_timeout`` to ``None`` to disable this behavior.
80  
81 .. _Pong frame: https://tools.ietf.org/html/rfc6455#section-5.5.3
82  
83 The ``close_timeout`` parameter defines a maximum wait time for completing
84 the closing handshake and terminating the TCP connection. For legacy
85 reasons, :meth:`close` completes in at most ``5 * close_timeout`` seconds.
86  
87 ``close_timeout`` needs to be a parameter of the protocol because
88 websockets usually calls :meth:`close` implicitly upon exit when
89 :func:`connect` is used as a context manager.
90  
91 To apply a timeout to any other API, wrap it in :func:`~asyncio.wait_for`.
92  
93 The ``max_size`` parameter enforces the maximum size for incoming messages
94 in bytes. The default value is 1 MiB. ``None`` disables the limit. If a
95 message larger than the maximum size is received, :meth:`recv` will
96 raise :exc:`~websockets.exceptions.ConnectionClosedError` and the
97 connection will be closed with code 1009.
98  
99 The ``max_queue`` parameter sets the maximum length of the queue that
100 holds incoming messages. The default value is ``32``. ``None`` disables
101 the limit. Messages are added to an in-memory queue when they're received;
102 then :meth:`recv` pops from that queue. In order to prevent excessive
103 memory consumption when messages are received faster than they can be
104 processed, the queue must be bounded. If the queue fills up, the protocol
105 stops processing incoming data until :meth:`recv` is called. In this
106 situation, various receive buffers (at least in :mod:`asyncio` and in the
107 OS) will fill up, then the TCP receive window will shrink, slowing down
108 transmission to avoid packet loss.
109  
110 Since Python can use up to 4 bytes of memory to represent a single
111 character, each connection may use up to ``4 * max_size * max_queue``
112 bytes of memory to store incoming messages. By default, this is 128 MiB.
113 You may want to lower the limits, depending on your application's
114 requirements.
115  
116 The ``read_limit`` argument sets the high-water limit of the buffer for
117 incoming bytes. The low-water limit is half the high-water limit. The
118 default value is 64 KiB, half of asyncio's default (based on the current
119 implementation of :class:`~asyncio.StreamReader`).
120  
121 The ``write_limit`` argument sets the high-water limit of the buffer for
122 outgoing bytes. The low-water limit is a quarter of the high-water limit.
123 The default value is 64 KiB, equal to asyncio's default (based on the
124 current implementation of ``FlowControlMixin``).
125  
126 As soon as the HTTP request and response in the opening handshake are
127 processed:
128  
129 * the request path is available in the :attr:`path` attribute;
130 * the request and response HTTP headers are available in the
131 :attr:`request_headers` and :attr:`response_headers` attributes,
132 which are :class:`~websockets.http.Headers` instances.

136  
137 Once the connection is closed, the code is available in the
138 :attr:`close_code` attribute and the reason in :attr:`close_reason`.
139  
140 All attributes must be treated as read-only.

409  
410 :func:`connect` can also be used as a asynchronous context manager::

414

416

420
