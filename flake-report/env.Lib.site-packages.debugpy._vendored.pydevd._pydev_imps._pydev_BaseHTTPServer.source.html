117 """HTTP request handler base class.
118  
119 The following explanation of HTTP serves to guide you through the
120 code as well as to expose any misunderstandings I may have about
121 HTTP (so you don't need to read the code to figure out I'm wrong
122 :-).
123  
124 HTTP (HyperText Transfer Protocol) is an extensible protocol on
125 top of a reliable stream transport (e.g. TCP/IP). The protocol
126 recognizes three parts to a request:
127  
128 1. One line identifying the request type and path
129 2. An optional set of RFC-822-style headers
130 3. An optional data part
131  
132 The headers and data are separated by a blank line.
133  
134 The first line of the request has the form
135  
136 <command> <path> <version>
137  
138 where <command> is a (case-sensitive) keyword such as GET or POST,
139 <path> is a string containing path information for the request,
140 and <version> should be the string "HTTP/1.0" or "HTTP/1.1".
141 <path> is encoded using the URL encoding scheme (using %xx to signify
142 the ASCII character with hex code xx).
143  
144 The specification specifies that lines are separated by CRLF but
145 for compatibility with the widest range of clients recommends
146 servers also handle LF. Similarly, whitespace in the request line
147 is treated sensibly (allowing multiple spaces between components
148 and allowing trailing whitespace).
149  
150 Similarly, for output, lines ought to be separated by CRLF pairs
151 but most clients grok LF characters just fine.
152  
153 If the first line of the request has the form
154  
155 <command> <path>
156  
157 (i.e. <version> is left out) then this is assumed to be an HTTP
158 0.9 request; this form has no optional headers and data part and
159 the reply consists of just the data.
160  
161 The reply form of the HTTP 1.x protocol again has three parts:
162  
163 1. One line giving the response code
164 2. An optional set of RFC-822-style headers
165 3. The data
166  
167 Again, the headers and data are separated by a blank line.
168  
169 The response code line has the form
170  
171 <version> <responsecode> <responsestring>
172  
173 where <version> is the protocol version ("HTTP/1.0" or "HTTP/1.1"),
174 <responsecode> is a 3-digit response code indicating success or
175 failure of the request, and <responsestring> is an optional
176 human-readable string explaining what the response code means.
177  
178 This server parses the request and the headers, and then calls a
179 function specific to the request type (<command>). Specifically,
180 a request SPAM will be handled by a method do_SPAM(). If no
181 such method exists the server sends an error response to the
182 client. If it exists, it is called with no arguments:
183  
184 do_SPAM()
185  
186 Note that the request name is case sensitive (i.e. SPAM and spam
187 are different requests).
188  
189 The various request details are stored in instance variables:
190  
191 - client_address is the client IP address in the form (host,
192 port);
193  
194 - command, path and version are the broken-down request line;
195  
196 - headers is an instance of mimetools.Message (or a derived
197 class) containing the header information;
198  
199 - rfile is a file object open for reading positioned at the
200 start of the optional input data part;
201  
202 - wfile is a file object open for writing.
203  
204 IT IS IMPORTANT TO ADHERE TO THE PROTOCOL FOR WRITING!
205  
206 The first thing to be written must be the response line. Then
207 follow 0 or more header lines, then a blank line, and then the
208 actual data (if any). The meaning of the header lines depends on
209 the command executed by the server; in most cases, when data is
210 returned, there should be at least one header line of the form
211  
212 Content-type: <type>/<subtype>
213  
214 where <type> and <subtype> should be registered MIME types,
215 e.g. "text/html" or "text/plain".
