14 Sequences/SERIAL/IDENTITY
15 -------------------------
16  
17 PostgreSQL supports sequences, and SQLAlchemy uses these as the default means
18 of creating new primary key values for integer-based primary key columns. When
19 creating tables, SQLAlchemy will issue the ``SERIAL`` datatype for
20 integer-based primary key columns, which generates a sequence and server side
21 default corresponding to the column.
22  
23 To specify a specific named sequence to be used for primary key generation,
24 use the :func:`~sqlalchemy.schema.Sequence` construct::
25  
26 Table('sometable', metadata,
27 Column('id', Integer, Sequence('some_id_seq'), primary_key=True)
28 )
29  
30 When SQLAlchemy issues a single INSERT statement, to fulfill the contract of
31 having the "last insert identifier" available, a RETURNING clause is added to
32 the INSERT statement which specifies the primary key columns should be
33 returned after the statement completes. The RETURNING functionality only takes
34 place if PostgreSQL 8.2 or later is in use. As a fallback approach, the
35 sequence, whether specified explicitly or implicitly via ``SERIAL``, is
36 executed independently beforehand, the returned value to be used in the
37 subsequent insert. Note that when an
38 :func:`~sqlalchemy.sql.expression.insert()` construct is executed using
39 "executemany" semantics, the "last inserted identifier" functionality does not
40 apply; no RETURNING clause is emitted nor is the sequence pre-executed in this
41 case.
42  
43 To force the usage of RETURNING by default off, specify the flag
44 ``implicit_returning=False`` to :func:`_sa.create_engine`.
45  
46 PostgreSQL 10 IDENTITY columns
47 ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
48  
49 PostgreSQL 10 has a new IDENTITY feature that supersedes the use of SERIAL.
50 Built-in support for rendering of IDENTITY is not available yet, however the
51 following compilation hook may be used to replace occurrences of SERIAL with
52 IDENTITY::
53  
54 from sqlalchemy.schema import CreateColumn
55 from sqlalchemy.ext.compiler import compiles
56  
57  
58 @compiles(CreateColumn, 'postgresql')
59 def use_identity(element, compiler, **kw):
60 text = compiler.visit_create_column(element, **kw)
61 text = text.replace("SERIAL", "INT GENERATED BY DEFAULT AS IDENTITY")
62 return text
63  
64 Using the above, a table such as::
65  
66 t = Table(
67 't', m,
68 Column('id', Integer, primary_key=True),
69 Column('data', String)
70 )
71  
72 Will generate on the backing database as::
73  
74 CREATE TABLE t (
75 id INT GENERATED BY DEFAULT AS IDENTITY NOT NULL,
76 data VARCHAR,
77 PRIMARY KEY (id)
78 )

82 Transaction Isolation Level
83 ---------------------------
84  
85 Most SQLAlchemy dialects support setting of transaction isolation level
86 using the :paramref:`_sa.create_engine.execution_options` parameter
87 at the :func:`_sa.create_engine` level, and at the :class:`_engine.Connection`
88 level via the :paramref:`.Connection.execution_options.isolation_level`
89 parameter.
90  
91 For PostgreSQL dialects, this feature works either by making use of the
92 DBAPI-specific features, such as psycopg2's isolation level flags which will
93 embed the isolation level setting inline with the ``"BEGIN"`` statement, or for
94 DBAPIs with no direct support by emitting ``SET SESSION CHARACTERISTICS AS
95 TRANSACTION ISOLATION LEVEL <level>`` ahead of the ``"BEGIN"`` statement
96 emitted by the DBAPI. For the special AUTOCOMMIT isolation level,
97 DBAPI-specific techniques are used which is typically an ``.autocommit``
98 flag on the DBAPI connection object.
99  
100 To set isolation level using :func:`_sa.create_engine`::
101  
102 engine = create_engine(
103 "postgresql+pg8000://scott:tiger@localhost/test",
104 execution_options={
105 "isolation_level": "REPEATABLE READ"
106 }
107 )
108  
109 To set using per-connection execution options::
110  
111 with engine.connect() as conn:
112 conn = conn.execution_options(
113 isolation_level="REPEATABLE READ"
114 )
115 with conn.begin():
116 # ... work with transaction
117  
118 Valid values for ``isolation_level`` on most PostgreSQL dialects include:
119  
120 * ``READ COMMITTED``
121 * ``READ UNCOMMITTED``
122 * ``REPEATABLE READ``
123 * ``SERIALIZABLE``
124 * ``AUTOCOMMIT``

136 Remote-Schema Table Introspection and PostgreSQL search_path
137 ------------------------------------------------------------
138  
139 **TL;DR;**: keep the ``search_path`` variable set to its default of ``public``,
140 name schemas **other** than ``public`` explicitly within ``Table`` definitions.
141  
142 The PostgreSQL dialect can reflect tables from any schema. The
143 :paramref:`_schema.Table.schema` argument, or alternatively the
144 :paramref:`.MetaData.reflect.schema` argument determines which schema will
145 be searched for the table or tables. The reflected :class:`_schema.Table`
146 objects
147 will in all cases retain this ``.schema`` attribute as was specified.
148 However, with regards to tables which these :class:`_schema.Table`
149 objects refer to
150 via foreign key constraint, a decision must be made as to how the ``.schema``
151 is represented in those remote tables, in the case where that remote
152 schema name is also a member of the current
153 `PostgreSQL search path
154 <http://www.postgresql.org/docs/current/static/ddl-schemas.html#DDL-SCHEMAS-PATH>`_.
155  
156 By default, the PostgreSQL dialect mimics the behavior encouraged by
157 PostgreSQL's own ``pg_get_constraintdef()`` builtin procedure. This function
158 returns a sample definition for a particular foreign key constraint,
159 omitting the referenced schema name from that definition when the name is
160 also in the PostgreSQL schema search path. The interaction below
161 illustrates this behavior::
162  
163 test=> CREATE TABLE test_schema.referred(id INTEGER PRIMARY KEY);
164 CREATE TABLE
165 test=> CREATE TABLE referring(
166 test(> id INTEGER PRIMARY KEY,
167 test(> referred_id INTEGER REFERENCES test_schema.referred(id));
168 CREATE TABLE
169 test=> SET search_path TO public, test_schema;
170 test=> SELECT pg_catalog.pg_get_constraintdef(r.oid, true) FROM
171 test-> pg_catalog.pg_class c JOIN pg_catalog.pg_namespace n
172 test-> ON n.oid = c.relnamespace
173 test-> JOIN pg_catalog.pg_constraint r ON c.oid = r.conrelid
174 test-> WHERE c.relname='referring' AND r.contype = 'f'
175 test-> ;
176 pg_get_constraintdef
177 ---------------------------------------------------
178 FOREIGN KEY (referred_id) REFERENCES referred(id)
179 (1 row)
180  
181 Above, we created a table ``referred`` as a member of the remote schema
182 ``test_schema``, however when we added ``test_schema`` to the
183 PG ``search_path`` and then asked ``pg_get_constraintdef()`` for the
184 ``FOREIGN KEY`` syntax, ``test_schema`` was not included in the output of
185 the function.
186  
187 On the other hand, if we set the search path back to the typical default
188 of ``public``::
189  
190 test=> SET search_path TO public;
191 SET
192  
193 The same query against ``pg_get_constraintdef()`` now returns the fully
194 schema-qualified name for us::
195  
196 test=> SELECT pg_catalog.pg_get_constraintdef(r.oid, true) FROM
197 test-> pg_catalog.pg_class c JOIN pg_catalog.pg_namespace n
198 test-> ON n.oid = c.relnamespace
199 test-> JOIN pg_catalog.pg_constraint r ON c.oid = r.conrelid
200 test-> WHERE c.relname='referring' AND r.contype = 'f';
201 pg_get_constraintdef
202 ---------------------------------------------------------------
203 FOREIGN KEY (referred_id) REFERENCES test_schema.referred(id)
204 (1 row)
205  
206 SQLAlchemy will by default use the return value of ``pg_get_constraintdef()``
207 in order to determine the remote schema name. That is, if our ``search_path``
208 were set to include ``test_schema``, and we invoked a table
209 reflection process as follows::
210  
211 >>> from sqlalchemy import Table, MetaData, create_engine
212 >>> engine = create_engine("postgresql://scott:tiger@localhost/test")
213 >>> with engine.connect() as conn:
214 ... conn.execute("SET search_path TO test_schema, public")
215 ... meta = MetaData()
216 ... referring = Table('referring', meta,
217 ... autoload=True, autoload_with=conn)
218 ...
219 <sqlalchemy.engine.result.ResultProxy object at 0x101612ed0>
220  
221 The above process would deliver to the :attr:`_schema.MetaData.tables`
222 collection
223 ``referred`` table named **without** the schema::
224  
225 >>> meta.tables['referred'].schema is None
226 True
227  
228 To alter the behavior of reflection such that the referred schema is
229 maintained regardless of the ``search_path`` setting, use the
230 ``postgresql_ignore_search_path`` option, which can be specified as a
231 dialect-specific argument to both :class:`_schema.Table` as well as
232 :meth:`_schema.MetaData.reflect`::
233  
234 >>> with engine.connect() as conn:
235 ... conn.execute("SET search_path TO test_schema, public")
236 ... meta = MetaData()
237 ... referring = Table('referring', meta, autoload=True,
238 ... autoload_with=conn,
239 ... postgresql_ignore_search_path=True)
240 ...
241 <sqlalchemy.engine.result.ResultProxy object at 0x1016126d0>
242  
243 We will now have ``test_schema.referred`` stored as schema-qualified::
244  
245 >>> meta.tables['test_schema.referred'].schema
246 'test_schema'

248 .. sidebar:: Best Practices for PostgreSQL Schema reflection
249  
250 The description of PostgreSQL schema reflection behavior is complex, and
251 is the product of many years of dealing with widely varied use cases and
252 user preferences. But in fact, there's no need to understand any of it if
253 you just stick to the simplest use pattern: leave the ``search_path`` set
254 to its default of ``public`` only, never refer to the name ``public`` as
255 an explicit schema name otherwise, and refer to all other schema names
256 explicitly when building up a :class:`_schema.Table` object. The options
257 described here are only for those users who can't, or prefer not to, stay
258 within these guidelines.
259  
260 Note that **in all cases**, the "default" schema is always reflected as
261 ``None``. The "default" schema on PostgreSQL is that which is returned by the
262 PostgreSQL ``current_schema()`` function. On a typical PostgreSQL
263 installation, this is the name ``public``. So a table that refers to another
264 which is in the ``public`` (i.e. default) schema will always have the
265 ``.schema`` attribute set to ``None``.
266

268 dialect-level option accepted by :class:`_schema.Table` and
269 :meth:`_schema.MetaData.reflect`.

278 INSERT/UPDATE...RETURNING
279 -------------------------
280  
281 The dialect supports PG 8.2's ``INSERT..RETURNING``, ``UPDATE..RETURNING`` and
282 ``DELETE..RETURNING`` syntaxes. ``INSERT..RETURNING`` is used by default
283 for single-row INSERT statements in order to fetch newly generated
284 primary key identifiers. To specify an explicit ``RETURNING`` clause,
285 use the :meth:`._UpdateBase.returning` method on a per-statement basis::
286  
287 # INSERT..RETURNING
288 result = table.insert().returning(table.c.col1, table.c.col2).\
289 values(name='foo')
290 print(result.fetchall())
291  
292 # UPDATE..RETURNING
293 result = table.update().returning(table.c.col1, table.c.col2).\
294 where(table.c.name=='foo').values(name='bar')
295 print(result.fetchall())
296  
297 # DELETE..RETURNING
298 result = table.delete().returning(table.c.col1, table.c.col2).\
299 where(table.c.name=='foo')
300 print(result.fetchall())

302 .. _postgresql_insert_on_conflict:
303  
304 INSERT...ON CONFLICT (Upsert)
305 ------------------------------
306  
307 Starting with version 9.5, PostgreSQL allows "upserts" (update or insert) of
308 rows into a table via the ``ON CONFLICT`` clause of the ``INSERT`` statement. A
309 candidate row will only be inserted if that row does not violate any unique
310 constraints. In the case of a unique constraint violation, a secondary action
311 can occur which can be either "DO UPDATE", indicating that the data in the
312 target row should be updated, or "DO NOTHING", which indicates to silently skip
313 this row.
314  
315 Conflicts are determined using existing unique constraints and indexes. These
316 constraints may be identified either using their name as stated in DDL,
317 or they may be *inferred* by stating the columns and conditions that comprise
318 the indexes.
319  
320 SQLAlchemy provides ``ON CONFLICT`` support via the PostgreSQL-specific
321 :func:`_postgresql.insert()` function, which provides
322 the generative methods :meth:`~.postgresql.Insert.on_conflict_do_update`
323 and :meth:`~.postgresql.Insert.on_conflict_do_nothing`::
324  
325 from sqlalchemy.dialects.postgresql import insert
326  
327 insert_stmt = insert(my_table).values(
328 id='some_existing_id',
329 data='inserted value')
330  
331 do_nothing_stmt = insert_stmt.on_conflict_do_nothing(
332 index_elements=['id']
333 )
334  
335 conn.execute(do_nothing_stmt)
336  
337 do_update_stmt = insert_stmt.on_conflict_do_update(
338 constraint='pk_my_table',
339 set_=dict(data='updated value')
340 )
341  
342 conn.execute(do_update_stmt)
343  
344 Both methods supply the "target" of the conflict using either the
345 named constraint or by column inference:
346  
347 * The :paramref:`.Insert.on_conflict_do_update.index_elements` argument
348 specifies a sequence containing string column names, :class:`_schema.Column`
349 objects, and/or SQL expression elements, which would identify a unique
350 index::
351  
352 do_update_stmt = insert_stmt.on_conflict_do_update(
353 index_elements=['id'],
354 set_=dict(data='updated value')
355 )
356  
357 do_update_stmt = insert_stmt.on_conflict_do_update(
358 index_elements=[my_table.c.id],
359 set_=dict(data='updated value')
360 )
361  
362 * When using :paramref:`.Insert.on_conflict_do_update.index_elements` to
363 infer an index, a partial index can be inferred by also specifying the
364 use the :paramref:`.Insert.on_conflict_do_update.index_where` parameter::
365  
366 from sqlalchemy.dialects.postgresql import insert
367  
368 stmt = insert(my_table).values(user_email='a@b.com', data='inserted data')
369 stmt = stmt.on_conflict_do_update(
370 index_elements=[my_table.c.user_email],
371 index_where=my_table.c.user_email.like('%@gmail.com'),
372 set_=dict(data=stmt.excluded.data)
373 )
374 conn.execute(stmt)
375  
376 * The :paramref:`.Insert.on_conflict_do_update.constraint` argument is
377 used to specify an index directly rather than inferring it. This can be
378 the name of a UNIQUE constraint, a PRIMARY KEY constraint, or an INDEX::
379  
380 do_update_stmt = insert_stmt.on_conflict_do_update(
381 constraint='my_table_idx_1',
382 set_=dict(data='updated value')
383 )
384  
385 do_update_stmt = insert_stmt.on_conflict_do_update(
386 constraint='my_table_pk',
387 set_=dict(data='updated value')
388 )
389  
390 * The :paramref:`.Insert.on_conflict_do_update.constraint` argument may
391 also refer to a SQLAlchemy construct representing a constraint,
392 e.g. :class:`.UniqueConstraint`, :class:`.PrimaryKeyConstraint`,
393 :class:`.Index`, or :class:`.ExcludeConstraint`. In this use,
394 if the constraint has a name, it is used directly. Otherwise, if the
395 constraint is unnamed, then inference will be used, where the expressions
396 and optional WHERE clause of the constraint will be spelled out in the
397 construct. This use is especially convenient
398 to refer to the named or unnamed primary key of a :class:`_schema.Table`
399 using the
400 :attr:`_schema.Table.primary_key` attribute::
401  
402 do_update_stmt = insert_stmt.on_conflict_do_update(
403 constraint=my_table.primary_key,
404 set_=dict(data='updated value')
405 )
406  
407 ``ON CONFLICT...DO UPDATE`` is used to perform an update of the already
408 existing row, using any combination of new values as well as values
409 from the proposed insertion. These values are specified using the
410 :paramref:`.Insert.on_conflict_do_update.set_` parameter. This
411 parameter accepts a dictionary which consists of direct values
412 for UPDATE::
413  
414 from sqlalchemy.dialects.postgresql import insert
415  
416 stmt = insert(my_table).values(id='some_id', data='inserted value')
417 do_update_stmt = stmt.on_conflict_do_update(
418 index_elements=['id'],
419 set_=dict(data='updated value')
420 )
421 conn.execute(do_update_stmt)
422  
423 .. warning::
424  
425 The :meth:`_expression.Insert.on_conflict_do_update`
426 method does **not** take into
427 account Python-side default UPDATE values or generation functions, e.g.
428 those specified using :paramref:`_schema.Column.onupdate`.
429 These values will not be exercised for an ON CONFLICT style of UPDATE,
430 unless they are manually specified in the
431 :paramref:`.Insert.on_conflict_do_update.set_` dictionary.
432  
433 In order to refer to the proposed insertion row, the special alias
434 :attr:`~.postgresql.Insert.excluded` is available as an attribute on
435 the :class:`_postgresql.Insert` object; this object is a
436 :class:`_expression.ColumnCollection`
437 which alias contains all columns of the target
438 table::
439  
440 from sqlalchemy.dialects.postgresql import insert
441  
442 stmt = insert(my_table).values(
443 id='some_id',
444 data='inserted value',
445 author='jlh')
446 do_update_stmt = stmt.on_conflict_do_update(
447 index_elements=['id'],
448 set_=dict(data='updated value', author=stmt.excluded.author)
449 )
450 conn.execute(do_update_stmt)
451  
452 The :meth:`_expression.Insert.on_conflict_do_update` method also accepts
453 a WHERE clause using the :paramref:`.Insert.on_conflict_do_update.where`
454 parameter, which will limit those rows which receive an UPDATE::
455  
456 from sqlalchemy.dialects.postgresql import insert
457  
458 stmt = insert(my_table).values(
459 id='some_id',
460 data='inserted value',
461 author='jlh')
462 on_update_stmt = stmt.on_conflict_do_update(
463 index_elements=['id'],
464 set_=dict(data='updated value', author=stmt.excluded.author)
465 where=(my_table.c.status == 2)
466 )
467 conn.execute(on_update_stmt)
468  
469 ``ON CONFLICT`` may also be used to skip inserting a row entirely
470 if any conflict with a unique or exclusion constraint occurs; below
471 this is illustrated using the
472 :meth:`~.postgresql.Insert.on_conflict_do_nothing` method::
473  
474 from sqlalchemy.dialects.postgresql import insert
475  
476 stmt = insert(my_table).values(id='some_id', data='inserted value')
477 stmt = stmt.on_conflict_do_nothing(index_elements=['id'])
478 conn.execute(stmt)
479  
480 If ``DO NOTHING`` is used without specifying any columns or constraint,
481 it has the effect of skipping the INSERT for any unique or exclusion
482 constraint violation which occurs::
483  
484 from sqlalchemy.dialects.postgresql import insert
485  
486 stmt = insert(my_table).values(id='some_id', data='inserted value')
487 stmt = stmt.on_conflict_do_nothing()
488 conn.execute(stmt)

500 Full Text Search
501 ----------------
502  
503 SQLAlchemy makes available the PostgreSQL ``@@`` operator via the
504 :meth:`_expression.ColumnElement.match`
505 method on any textual column expression.
506 On a PostgreSQL dialect, an expression like the following::
507  
508 select([sometable.c.text.match("search string")])
509  
510 will emit to the database::
511  
512 SELECT text @@ to_tsquery('search string') FROM table
513  
514 The PostgreSQL text search functions such as ``to_tsquery()``
515 and ``to_tsvector()`` are available
516 explicitly using the standard :data:`.func` construct. For example::
517  
518 select([
519 func.to_tsvector('fat cats ate rats').match('cat & rat')
520 ])
521  
522 Emits the equivalent of::
523  
524 SELECT to_tsvector('fat cats ate rats') @@ to_tsquery('cat & rat')
525  
526 The :class:`_postgresql.TSVECTOR` type can provide for explicit CAST::
527  
528 from sqlalchemy.dialects.postgresql import TSVECTOR
529 from sqlalchemy import select, cast
530 select([cast("some text", TSVECTOR)])
531  
532 produces a statement equivalent to::
533  
534 SELECT CAST('some text' AS TSVECTOR) AS anon_1
535  
536 Full Text Searches in PostgreSQL are influenced by a combination of: the
537 PostgreSQL setting of ``default_text_search_config``, the ``regconfig`` used
538 to build the GIN/GiST indexes, and the ``regconfig`` optionally passed in
539 during a query.
540  
541 When performing a Full Text Search against a column that has a GIN or
542 GiST index that is already pre-computed (which is common on full text
543 searches) one may need to explicitly pass in a particular PostgreSQL
544 ``regconfig`` value to ensure the query-planner utilizes the index and does
545 not re-compute the column on demand.
546  
547 In order to provide for this explicit query planning, or to use different
548 search strategies, the ``match`` method accepts a ``postgresql_regconfig``
549 keyword argument::
550  
551 select([mytable.c.id]).where(
552 mytable.c.title.match('somestring', postgresql_regconfig='english')
553 )
554  
555 Emits the equivalent of::
556  
557 SELECT mytable.id FROM mytable
558 WHERE mytable.title @@ to_tsquery('english', 'somestring')
559  
560 One can also specifically pass in a `'regconfig'` value to the
561 ``to_tsvector()`` command as the initial argument::
562  
563 select([mytable.c.id]).where(
564 func.to_tsvector('english', mytable.c.title )\
565 .match('somestring', postgresql_regconfig='english')
566 )
567  
568 produces a statement equivalent to::
569  
570 SELECT mytable.id FROM mytable
571 WHERE to_tsvector('english', mytable.title) @@
572 to_tsquery('english', 'somestring')
573  
574 It is recommended that you use the ``EXPLAIN ANALYZE...`` tool from
575 PostgreSQL to ensure that you are generating queries with SQLAlchemy that
576 take full advantage of any indexes you may have created for full text search.

578 FROM ONLY ...
579 -------------
580

592 dialect_name='postgresql')

606 .. _postgresql_partial_indexes:

608 Partial Indexes
609 ^^^^^^^^^^^^^^^
610  
611 Partial indexes add criterion to the index definition so that the index is
612 applied to a subset of rows. These can be specified on :class:`.Index`
613 using the ``postgresql_where`` keyword argument::
614  
615 Index('my_index', my_table.c.id, postgresql_where=my_table.c.value > 10)
616  
617 .. _postgresql_operator_classes:
618  
619 Operator Classes
620 ^^^^^^^^^^^^^^^^
621  
622 PostgreSQL allows the specification of an *operator class* for each column of
623 an index (see
624 http://www.postgresql.org/docs/8.3/interactive/indexes-opclass.html).
625 The :class:`.Index` construct allows these to be specified via the
626 ``postgresql_ops`` keyword argument::
627  
628 Index(
629 'my_index', my_table.c.id, my_table.c.data,
630 postgresql_ops={
631 'data': 'text_pattern_ops',
632 'id': 'int4_ops'
633 })
634  
635 Note that the keys in the ``postgresql_ops`` dictionaries are the
636 "key" name of the :class:`_schema.Column`, i.e. the name used to access it from
637 the ``.c`` collection of :class:`_schema.Table`, which can be configured to be
638 different than the actual name of the column as expressed in the database.
639  
640 If ``postgresql_ops`` is to be used against a complex SQL expression such
641 as a function call, then to apply to the column it must be given a label
642 that is identified in the dictionary by name, e.g.::
643  
644 Index(
645 'my_index', my_table.c.id,
646 func.lower(my_table.c.data).label('data_lower'),
647 postgresql_ops={
648 'data_lower': 'text_pattern_ops',
649 'id': 'int4_ops'
650 })
651  
652 Operator classes are also supported by the

654 :paramref:`_postgresql.ExcludeConstraint.ops` parameter. See that parameter for
655 details.

658 :class:`_postgresql.ExcludeConstraint`.

661 Index Types
662 ^^^^^^^^^^^
663  
664 PostgreSQL provides several index types: B-Tree, Hash, GiST, and GIN, as well
665 as the ability for users to create their own (see
666 http://www.postgresql.org/docs/8.3/static/indexes-types.html). These can be
667 specified on :class:`.Index` using the ``postgresql_using`` keyword argument::
668  
669 Index('my_index', my_table.c.data, postgresql_using='gin')
670  
671 The value passed to the keyword argument will be simply passed through to the
672 underlying CREATE INDEX command, so it *must* be a valid index type for your
673 version of PostgreSQL.

675 .. _postgresql_index_storage:
676  
677 Index Storage Parameters
678 ^^^^^^^^^^^^^^^^^^^^^^^^
679  
680 PostgreSQL allows storage parameters to be set on indexes. The storage
681 parameters available depend on the index method used by the index. Storage
682 parameters can be specified on :class:`.Index` using the ``postgresql_with``
683 keyword argument::
684  
685 Index('my_index', my_table.c.data, postgresql_with={"fillfactor": 50})
686  
687 .. versionadded:: 1.0.6
688  
689 PostgreSQL allows to define the tablespace in which to create the index.
690 The tablespace can be specified on :class:`.Index` using the
691 ``postgresql_tablespace`` keyword argument::
692  
693 Index('my_index', my_table.c.data, postgresql_tablespace='my_tablespace')
694  
695 .. versionadded:: 1.1
696  
697 Note that the same option is available on :class:`_schema.Table` as well.

699 .. _postgresql_index_concurrently:
700  
701 Indexes with CONCURRENTLY
702 ^^^^^^^^^^^^^^^^^^^^^^^^^
703  
704 The PostgreSQL index option CONCURRENTLY is supported by passing the
705 flag ``postgresql_concurrently`` to the :class:`.Index` construct::
706  
707 tbl = Table('testtbl', m, Column('data', Integer))
708  
709 idx1 = Index('test_idx1', tbl.c.data, postgresql_concurrently=True)
710  
711 The above index construct will render DDL for CREATE INDEX, assuming
712 PostgreSQL 8.2 or higher is detected or for a connection-less dialect, as::
713  
714 CREATE INDEX CONCURRENTLY test_idx1 ON testtbl (data)
715  
716 For DROP INDEX, assuming PostgreSQL 9.2 or higher is detected or for
717 a connection-less dialect, it will emit::
718  
719 DROP INDEX CONCURRENTLY test_idx1
720  
721 .. versionadded:: 1.1 support for CONCURRENTLY on DROP INDEX. The
722 CONCURRENTLY keyword is now only emitted if a high enough version
723 of PostgreSQL is detected on the connection (or for a connection-less
724 dialect).
725  
726 When using CONCURRENTLY, the PostgreSQL database requires that the statement
727 be invoked outside of a transaction block. The Python DBAPI enforces that
728 even for a single statement, a transaction is present, so to use this
729 construct, the DBAPI's "autocommit" mode must be used::
730  
731 metadata = MetaData()
732 table = Table(
733 "foo", metadata,
734 Column("id", String))
735 index = Index(
736 "foo_idx", table.c.id, postgresql_concurrently=True)
737  
738 with engine.connect() as conn:
739 with conn.execution_options(isolation_level='AUTOCOMMIT'):
740 table.create(conn)
741  
742 .. seealso::
743  
744 :ref:`postgresql_isolation_level`

748 PostgreSQL Index Reflection
749 ---------------------------
750  
751 The PostgreSQL database creates a UNIQUE INDEX implicitly whenever the
752 UNIQUE CONSTRAINT construct is used. When inspecting a table using
753 :class:`_reflection.Inspector`, the :meth:`_reflection.Inspector.get_indexes`
754 and the :meth:`_reflection.Inspector.get_unique_constraints`
755 will report on these
756 two constructs distinctly; in the case of the index, the key
757 ``duplicates_constraint`` will be present in the index entry if it is
758 detected as mirroring a constraint. When performing reflection using
759 ``Table(..., autoload=True)``, the UNIQUE INDEX is **not** returned
760 in :attr:`_schema.Table.indexes` when it is detected as mirroring a
761 :class:`.UniqueConstraint` in the :attr:`_schema.Table.constraints` collection
762 .

767 collection; the PostgreSQL backend will no longer include a "mirrored"
768 :class:`.Index` construct in :attr:`_schema.Table.indexes`

930 .. _postgresql_array_of_enum:
931  
932 Using ENUM with ARRAY
933 ^^^^^^^^^^^^^^^^^^^^^
934  
935 The combination of ENUM and ARRAY is not directly supported by backend
936 DBAPIs at this time. Prior to SQLAlchemy 1.3.17, a special workaround
937 was needed in order to allow this combination to work, described below.
938  
939 .. versionchanged:: 1.3.17 The combination of ENUM and ARRAY is now directly
940 handled by SQLAlchemy's implementation without any workarounds needed.
941  
942 .. sourcecode:: python
943  
944 from sqlalchemy import TypeDecorator
945 from sqlalchemy.dialects.postgresql import ARRAY
946

948 impl = ARRAY
949  
950 def bind_expression(self, bindvalue):
951 return sa.cast(bindvalue, self)
952

955 dialect, coltype)
956

958 inner = re.match(r"^{(.*)}$", value).group(1)
959 return inner.split(",") if inner else []
960  
961 def process(value):
962 if value is None:
963 return None
964 return super_rp(handle_raw_string(value))
965 return process
966  
967 E.g.::
968  
969 Table(
970 'mydata', metadata,
971 Column('id', Integer, primary_key=True),
972 Column('data', ArrayOfEnum(ENUM('a', 'b, 'c', name='myenum')))
973  
974 )
975  
976 This type is not included as a built-in type as it would be incompatible
977 with a DBAPI that suddenly decides to support ARRAY of ENUM directly in
978 a new version.

1440 table.create(engine) # will emit CREATE ENUM and CREATE TABLE
1441 table.drop(engine) # will emit DROP TABLE and DROP ENUM

1443 To use a common enumerated type between multiple tables, the best
1444 practice is to declare the :class:`_types.Enum` or
1445 :class:`_postgresql.ENUM` independently, and associate it with the
1446 :class:`_schema.MetaData` object itself::
1447  
1448 my_enum = ENUM('a', 'b', 'c', name='myenum', metadata=metadata)
1449  
1450 t1 = Table('sometable_one', metadata,
1451 Column('some_enum', myenum)
1452 )
1453  
1454 t2 = Table('sometable_two', metadata,
1455 Column('some_enum', myenum)
1456 )
1457  
1458 When this pattern is used, care must still be taken at the level
1459 of individual table creates. Emitting CREATE TABLE without also
1460 specifying ``checkfirst=True`` will still cause issues::
1461  
1462 t1.create(engine) # will fail: no such type 'myenum'
1463  
1464 If we specify ``checkfirst=True``, the individual table-level create
1465 operation will check for the ``ENUM`` and create if not exists::
1466  
1467 # will check if enum exists, and emit CREATE TYPE if not
1468 t1.create(engine, checkfirst=True)
1469  
1470 When using a metadata-level ENUM type, the type will always be created
1471 and dropped if either the metadata-wide create/drop is called::
1472  
1473 metadata.create_all(engine) # will emit CREATE TYPE
1474 metadata.drop_all(engine) # will emit DROP TYPE
1475  
1476 The type can also be created and dropped directly::
1477  
1478 my_enum.create(engine)
1479 my_enum.drop(engine)
1480  
1481 .. versionchanged:: 1.0.0 The PostgreSQL :class:`_postgresql.ENUM` type
1482 now behaves more strictly with regards to CREATE/DROP. A metadata-level
1483 ENUM type will only be created and dropped at the metadata level,
1484 not the table level, with the exception of
1485 ``table.create(checkfirst=True)``.
1486 The ``table.drop()`` call will now emit a DROP TYPE for a table-level
1487 enumerated type.
