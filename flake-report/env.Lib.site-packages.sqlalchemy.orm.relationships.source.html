163 """Provide a relationship between two mapped classes.
164  
165 This corresponds to a parent-child or associative table relationship.
166 The constructed class is an instance of
167 :class:`.RelationshipProperty`.
168  
169 A typical :func:`_orm.relationship`, used in a classical mapping::
170  
171 mapper(Parent, properties={
172 'children': relationship(Child)
173 })
174  
175 Some arguments accepted by :func:`_orm.relationship`
176 optionally accept a
177 callable function, which when called produces the desired value.
178 The callable is invoked by the parent :class:`_orm.Mapper` at "mapper
179 initialization" time, which happens only when mappers are first used,
180 and is assumed to be after all mappings have been constructed. This
181 can be used to resolve order-of-declaration and other dependency
182 issues, such as if ``Child`` is declared below ``Parent`` in the same
183 file::
184  
185 mapper(Parent, properties={
186 "children":relationship(lambda: Child,
187 order_by=lambda: Child.id)
188 })
189  
190 When using the :ref:`declarative_toplevel` extension, the Declarative
191 initializer allows string arguments to be passed to
192 :func:`_orm.relationship`. These string arguments are converted into
193 callables that evaluate the string as Python code, using the
194 Declarative class-registry as a namespace. This allows the lookup of
195 related classes to be automatic via their string name, and removes the
196 need for related classes to be imported into the local module space
197 before the dependent classes have been declared. It is still required
198 that the modules in which these related classes appear are imported
199 anywhere in the application at some point before the related mappings
200 are actually used, else a lookup error will be raised when the
201 :func:`_orm.relationship`
202 attempts to resolve the string reference to the
203 related class. An example of a string- resolved class is as
204 follows::
205  
206 from sqlalchemy.ext.declarative import declarative_base
207  
208 Base = declarative_base()
209  
210 class Parent(Base):
211 __tablename__ = 'parent'
212 id = Column(Integer, primary_key=True)
213 children = relationship("Child", order_by="Child.id")

579

582 fetch for simple many-to-one references.
583

596 requested.

599 are loaded, using one or more additional SQL statements, which

694 :param passive_deletes=False:
695 Indicates loading behavior during delete operations.
696  
697 A value of True indicates that unloaded child items should not
698 be loaded during a delete operation on the parent. Normally,
699 when a parent item is deleted, all child items are loaded so
700 that they can either be marked as deleted, or have their
701 foreign key to the parent set to NULL. Marking this flag as
702 True usually implies an ON DELETE <CASCADE|SET NULL> rule is in
703 place which will handle updating/deleting child rows on the
704 database side.
705  
706 Additionally, setting the flag to the string value 'all' will
707 disable the "nulling out" of the child foreign keys, when the parent
708 object is deleted and there is no delete or delete-orphan cascade
709 enabled. This is typically used when a triggering or error raise
710 scenario is in place on the database side. Note that the foreign
711 key attributes on in-session child objects will not be changed after
712 a flush occurs so this is a very special use-case setting.
713 Additionally, the "nulling out" will still occur if the child
714 object is de-associated with the parent.
