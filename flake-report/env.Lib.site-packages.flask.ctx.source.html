266 class RequestContext:
267 """The request context contains all request relevant information. It is
268 created at the beginning of the request and pushed to the
269 `_request_ctx_stack` and removed at the end of it. It will create the
270 URL adapter and request object for the WSGI environment provided.
271  
272 Do not attempt to use this class directly, instead use
273 :meth:`~flask.Flask.test_request_context` and
274 :meth:`~flask.Flask.request_context` to create this object.
275  
276 When the request context is popped, it will evaluate all the
277 functions registered on the application for teardown execution
278 (:meth:`~flask.Flask.teardown_request`).
279  
280 The request context is automatically popped at the end of the request
281 for you. In debug mode the request context is kept around if
282 exceptions happen so that interactive debuggers have a chance to
283 introspect the data. With 0.4 this can also be forced for requests
284 that did not fail and outside of ``DEBUG`` mode. By setting
285 ``'flask._preserve_context'`` to ``True`` on the WSGI environment the
286 context will not pop itself at the end of the request. This is used by
287 the :meth:`~flask.Flask.test_client` for example to implement the
288 deferred cleanup functionality.
289  
290 You might find this helpful for unittests where you need the
291 information from the context local around for a little longer. Make
292 sure to properly :meth:`~werkzeug.LocalStack.pop` the stack yourself in
293 that situation, otherwise your unittests will leak memory.
294 """
