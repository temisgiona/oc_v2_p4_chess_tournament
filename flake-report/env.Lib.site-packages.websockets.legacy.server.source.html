61 class WebSocketServerProtocol(WebSocketCommonProtocol):
62 """
63 :class:`~asyncio.Protocol` subclass implementing a WebSocket server.
64  
65 :class:`WebSocketServerProtocol`:
66  
67 * performs the opening handshake to establish the connection;
68 * provides :meth:`recv` and :meth:`send` coroutines for receiving and
69 sending messages;
70 * deals with control frames automatically;
71 * performs the closing handshake to terminate the connection.
72  
73 You may customize the opening handshake by subclassing
74 :class:`WebSocketServer` and overriding:
75  
76 * :meth:`process_request` to intercept the client request before any
77 processing and, if appropriate, to abort the WebSocket request and
78 return a HTTP response instead;
79 * :meth:`select_subprotocol` to select a subprotocol, if the client and
80 the server have multiple subprotocols in common and the default logic
81 for choosing one isn't suitable (this is rarely needed).
82  
83 :class:`WebSocketServerProtocol` supports asynchronous iteration::
84  
85 async for message in websocket:
86 await process(message)
87  
88 The iterator yields incoming messages. It exits normally when the
89 connection is closed with the close code 1000 (OK) or 1001 (going away).
90 It raises a :exc:`~websockets.exceptions.ConnectionClosedError` exception
91 when the connection is closed with any other code.
92  
93 Once the connection is open, a `Ping frame`_ is sent every
94 ``ping_interval`` seconds. This serves as a keepalive. It helps keeping
95 the connection open, especially in the presence of proxies with short
96 timeouts on inactive connections. Set ``ping_interval`` to ``None`` to
97 disable this behavior.
98  
99 .. _Ping frame: https://tools.ietf.org/html/rfc6455#section-5.5.2
100  
101 If the corresponding `Pong frame`_ isn't received within ``ping_timeout``
102 seconds, the connection is considered unusable and is closed with
103 code 1011. This ensures that the remote endpoint remains responsive. Set
104 ``ping_timeout`` to ``None`` to disable this behavior.
105  
106 .. _Pong frame: https://tools.ietf.org/html/rfc6455#section-5.5.3
107  
108 The ``close_timeout`` parameter defines a maximum wait time for completing
109 the closing handshake and terminating the TCP connection. For legacy
110 reasons, :meth:`close` completes in at most ``4 * close_timeout`` seconds.
111  
112 ``close_timeout`` needs to be a parameter of the protocol because
113 websockets usually calls :meth:`close` implicitly when the connection
114 handler terminates.
115  
116 To apply a timeout to any other API, wrap it in :func:`~asyncio.wait_for`.
117  
118 The ``max_size`` parameter enforces the maximum size for incoming messages
119 in bytes. The default value is 1 MiB. ``None`` disables the limit. If a
120 message larger than the maximum size is received, :meth:`recv` will
121 raise :exc:`~websockets.exceptions.ConnectionClosedError` and the
122 connection will be closed with code 1009.
123  
124 The ``max_queue`` parameter sets the maximum length of the queue that
125 holds incoming messages. The default value is ``32``. ``None`` disables
126 the limit. Messages are added to an in-memory queue when they're received;
127 then :meth:`recv` pops from that queue. In order to prevent excessive
128 memory consumption when messages are received faster than they can be
129 processed, the queue must be bounded. If the queue fills up, the protocol
130 stops processing incoming data until :meth:`recv` is called. In this
131 situation, various receive buffers (at least in :mod:`asyncio` and in the
132 OS) will fill up, then the TCP receive window will shrink, slowing down
133 transmission to avoid packet loss.
134  
135 Since Python can use up to 4 bytes of memory to represent a single
136 character, each connection may use up to ``4 * max_size * max_queue``
137 bytes of memory to store incoming messages. By default, this is 128 MiB.
138 You may want to lower the limits, depending on your application's
139 requirements.
140  
141 The ``read_limit`` argument sets the high-water limit of the buffer for
142 incoming bytes. The low-water limit is half the high-water limit. The
143 default value is 64 KiB, half of asyncio's default (based on the current
144 implementation of :class:`~asyncio.StreamReader`).
145  
146 The ``write_limit`` argument sets the high-water limit of the buffer for
147 outgoing bytes. The low-water limit is a quarter of the high-water limit.
148 The default value is 64 KiB, equal to asyncio's default (based on the
149 current implementation of ``FlowControlMixin``).
150  
151 As soon as the HTTP request and response in the opening handshake are
152 processed:
153  
154 * the request path is available in the :attr:`path` attribute;
155 * the request and response HTTP headers are available in the
156 :attr:`request_headers` and :attr:`response_headers` attributes,
157 which are :class:`~websockets.http.Headers` instances.
158  
159 If a subprotocol was negotiated, it's available in the :attr:`subprotocol`
160 attribute.
161  
162 Once the connection is closed, the code is available in the
163 :attr:`close_code` attribute and the reason in :attr:`close_reason`.
164  
165 All attributes must be treated as read-only.

789 async def _close(self) -> None:

792

794 :class:`~asyncio.Server` object to stop accepting new connections and
795 then closes open connections with close code 1001.
796

798 # Stop accepting new connections.
799 self.server.close()

803

805 # register(). See https://bugs.python.org/issue34852 for details.

807 0, loop=self.loop if sys.version_info[:2] < (3, 8) else None

822 )

831 )

857 class Serve:
858 """
859  
860 Create, start, and return a WebSocket server on ``host`` and ``port``.
861  
862 Whenever a client connects, the server accepts the connection, creates a
863 :class:`WebSocketServerProtocol`, performs the opening handshake, and
864 delegates to the connection handler defined by ``ws_handler``. Once the
865 handler completes, either normally or with an exception, the server
866 performs the closing handshake and closes the connection.
867  
868 Awaiting :func:`serve` yields a :class:`WebSocketServer`. This instance
869 provides :meth:`~WebSocketServer.close` and
870 :meth:`~WebSocketServer.wait_closed` methods for terminating the server
871 and cleaning up its resources.
872  
873 When a server is closed with :meth:`~WebSocketServer.close`, it closes all
874 connections with close code 1001 (going away). Connections handlers, which
875 are running the ``ws_handler`` coroutine, will receive a
876 :exc:`~websockets.exceptions.ConnectionClosedOK` exception on their
877 current or next interaction with the WebSocket connection.
878  
879 :func:`serve` can also be used as an asynchronous context manager::
880  
881 stop = asyncio.Future() # set this future to exit the server
882  
883 async with serve(...):
884 await stop
885  
886 In this case, the server is shut down when exiting the context.
887  
888 :func:`serve` is a wrapper around the event loop's
889 :meth:`~asyncio.loop.create_server` method. It creates and starts a
890 :class:`asyncio.Server` with :meth:`~asyncio.loop.create_server`. Then it
891 wraps the :class:`asyncio.Server` in a :class:`WebSocketServer` and
892 returns the :class:`WebSocketServer`.
893  
894 ``ws_handler`` is the WebSocket handler. It must be a coroutine accepting
895 two arguments: the WebSocket connection, which is an instance of
896 :class:`WebSocketServerProtocol`, and the path of the request.
897  
898 The ``host`` and ``port`` arguments, as well as unrecognized keyword
899 arguments, are passed to :meth:`~asyncio.loop.create_server`.
900  
901 For example, you can set the ``ssl`` keyword argument to a
902 :class:`~ssl.SSLContext` to enable TLS.
903  
904 ``create_protocol`` defaults to :class:`WebSocketServerProtocol`. It may
905 be replaced by a wrapper or a subclass to customize the protocol that
906 manages the connection.
907  
908 The behavior of ``ping_interval``, ``ping_timeout``, ``close_timeout``,
909 ``max_size``, ``max_queue``, ``read_limit``, and ``write_limit`` is
910 described in :class:`WebSocketServerProtocol`.
911  
912 :func:`serve` also accepts the following optional arguments:
913  
914 * ``compression`` is a shortcut to configure compression extensions;
915 by default it enables the "permessage-deflate" extension; set it to
916 ``None`` to disable compression.
917 * ``origins`` defines acceptable Origin HTTP headers; include ``None`` in
918 the list if the lack of an origin is acceptable.
919 * ``extensions`` is a list of supported extensions in order of
920 decreasing preference.
921 * ``subprotocols`` is a list of supported subprotocols in order of
922 decreasing preference.
923 * ``extra_headers`` sets additional HTTP response headers when the
924 handshake succeeds; it can be a :class:`~websockets.http.Headers`
925 instance, a :class:`~collections.abc.Mapping`, an iterable of ``(name,
926 value)`` pairs, or a callable taking the request path and headers in
927 arguments and returning one of the above.
928 * ``process_request`` allows intercepting the HTTP request; it must be a
929 coroutine taking the request path and headers in argument; see
930 :meth:`~WebSocketServerProtocol.process_request` for details.
931 * ``select_subprotocol`` allows customizing the logic for selecting a
932 subprotocol; it must be a callable taking the subprotocols offered by
933 the client and available on the server in argument; see
934 :meth:`~WebSocketServerProtocol.select_subprotocol` for details.
935  
936 Since there's no useful way to propagate exceptions triggered in handlers,
937 they're sent to the ``"websockets.server"`` logger instead.
938 Debugging is much easier if you configure logging to print them::
939  
940 import logging
941 logger = logging.getLogger("websockets.server")
942 logger.setLevel(logging.ERROR)
943 logger.addHandler(logging.StreamHandler())
